### Day 20：Promise：給未來的一個承諾，解決你的回呼地獄
- **核心關鍵詞**：Promise, Resolve, Reject, then, catch
- **RPG 比喻**：傳奇武器的「預約券」：既然 D19 說我們不等矮人打劍，那我們領張券隨時追蹤。
- **高手直覺 (React 伏筆)**：非同步資料流的標準處理 (網路請求的底層機制)。
- **發想心得**：

### Day 23：錯誤處理：使用 Try...Catch 讓你的網頁不會整頁白屏
- **核心關鍵詞**：try, catch, finally, throwable
- **RPG 比喻**：自動復活保險盾 (當法術失敗炸裂時，不讓勇者直接陣亡，而是傳回村莊治療)。
- **高手直覺 (React 伏筆)**：React Error Boundary 的技術雛形與抗打擊開發 (Robustness)。
- **發想心得**：
    1. 為什麼會炸裂？常見崩潰原因：變數未定義、API 連線失敗、JSON 解析錯誤。
    2. 三段式防禦：
        - `try`：冒著風險去執行任務。
        - `catch`：萬一炸了，趕快接住那個「錯誤物件」。
        - `finally`：不管成功失敗，最後都要「關掉 Loading 燈」。
    3. 優雅的道歉：不要只看 console，要學會畫一個「哎呀，出錯了」的畫面給使用者看。
    4. 盾牌放哪裡？判斷精準度：不是包越多越好，而是包在「最脆弱（如 fetch）」與「最能處理（如渲染 UI）」的地方。
- **期待目標**：
    1. 建立「防禦性編程」思維，主動捕捉可能出錯的代碼塊。
    2. 掌握 `finally` 在實務中清空狀態（如取消載入中圖標）的用法。
- **延伸教材**：
    1. 丟出你自己的炸彈：throw new Error()。
    2. 非同步與錯誤處理：為什麼 fetch 的 404 不會觸發 catch？(極高機率被問的面試點)。
    3. 實戰建議：該怎麼判斷 try 該放在哪？(局部包覆 vs 全域捕捉)。

### Day 25：效能神兵：防抖 (Debounce) 與節流 (Throttle)
